## やりたいこと

- 会議タイマー
- あらかじめ設定したプログラムに対応したタイマー動作
- 全体としての遅れを常に管理
- 遅れに応じてプログラムの時間を伸縮し全体としての時間を調整（CSSのflexboxの考え方に近い）
- たとえば、質疑応答を短くする/伸ばすなどの自動調整
- 各プログラムには以下を設定できる
  - 名称
  - 時間縮めない優先度(5段階)
  - 最小時間（秒）
  - 省略判断優先度(5段階)
  - 省略判断時間（秒）(これより短くなってしまうならプログラム自体を省略する)
  - 時間伸ばす優先度(5段階)
  - 最大時間（秒）
- 音・画面で発表者に時間を知らせる（閾値で警告）
- バスの運行管理などスケジューリング全般への応用を想定

## 設計メモ

### 概要
- 予定されたプログラム群を与えられた総時間内に収めつつ、優先度に基づき各プログラムの時間を自動で調整する高機能タイマー

### 基本方針
- 各プログラムに「基準時間（baseline）」「最小/最大時間」「優先度」を持たせ、全体の余裕/不足に応じて比率的に伸縮する。
- 縮めるべきか省略すべきかは優先度と閾値（省略判断時間）で決める。

### データモデル（JSON 例・単位: 秒）

```json
{
  "totalTime": 3600,
  "programs": [
    {
      "id": "p1",
      "name": "開会挨拶",
      "baseline": 300,
      "min": 180,
      "max": 600,
      "dontShrinkPriority": 5,
      "skipPriority": 1,
      "skipThreshold": 60,
      "extendPriority": 3
    },
    {
      "id": "p2",
      "name": "発表A",
      "baseline": 1200,
      "min": 600,
      "max": 1800,
      "dontShrinkPriority": 3,
      "skipPriority": 2,
      "skipThreshold": 120,
      "extendPriority": 2
    }
  ]
}
```

フィールド説明:
- `baseline`: 基準／予定時間（秒）
- `min` / `max`: 最小／最大許容時間（秒）
- `dontShrinkPriority`: 時間を縮めない優先度（1: 縮めて良い ～ 5: 縮めたくない）
- `skipPriority`: 省略優先度（1: 省略しにくい ～ 5: 優先的に省略する）
- `skipThreshold`: この秒数より短くなってしまうならプログラムを省略する
- `extendPriority`: 伸ばす優先度（1: 伸ばしにくい ～ 5: 優先的に伸ばす）

### 調整アルゴリズム（基本案）

記号や式はすべてインラインコードで表現します。各プログラムの現在割当時間を `current_time`、基準時間を `baseline`、総利用可能時間を `T_total` とします。全プログラムの合計を `S = sum(baseline_i for i in programs)` と定義します。

- 基本ケース: `S == T_total` のとき変更なし。

- オーバー（`S > T_total`）のとき（不足量 `D = S - T_total`）は、優先度レベル `5 -> 4 -> 3 -> 2 -> 1` の順で縮小（`dontShrinkPriority`）と省略（`skipPriority`）を繰り返して不足を埋めます。手順は次のとおりです。

1. レベル `p` を `5` から `1` までループします。

2. 【縮小フェーズ】
   - `G_shrink = programs.filter(item => item.dontShrinkPriority === p && (item.current_time - item.min) > 0)`
   - 各項目の縮小余地を `shrinkPotential = Math.max(0, item.current_time - item.min)` とする。
   - `G_shrink` が存在すれば、縮小戦略を選択して `D` を減らす。
     - 比率縮小（`shrinkStrategy = "proportional"`）の JavaScript 実装例:

```javascript
// G_shrink: programs.filter(...)
// D: 不足時間（秒）
const sumPotential = G_shrink.reduce((s, it) => s + Math.max(0, it.current_time - it.min), 0);
if (sumPotential > 0) {
  for (const item of G_shrink) {
    const shrinkPotential = Math.max(0, item.current_time - item.min);
    const delta = Math.min(shrinkPotential, D * (shrinkPotential / sumPotential));
    item.current_time -= delta;
    D -= delta;
    if (D <= 0) break;
  }
}
```

     - 順次縮小（`shrinkStrategy = "sequential"`）の JavaScript 実装例:

```javascript
for (const item of G_shrink) {
  const shrinkPotential = Math.max(0, item.current_time - item.min);
  const delta = Math.min(shrinkPotential, D);
  item.current_time -= delta;
  D -= delta;
  if (D <= 0) break;
}
```

   - 縮小を適用後、もし `D <= 0` になれば終了します。

3. 【省略フェーズ】（縮小でまだ `D > 0` の場合）
   - `G_skip = programs.filter(item => item.skipPriority === p && item.current_time > 0)`
   - `G_skip` を時系列の手前（先頭）から順に処理し、各項目を完全に省略 (`delta = item.current_time; item.current_time = 0`) して `D -= delta` を行う。
   - 省略の途中で `D <= 0` になった場合、**省略によって `S` が `T_total` より小さくなる（時間が余る）ことがある**。その場合は以下を行う:
     - `S = programs.reduce((s, it) => s + it.current_time, 0)` を再計算し、余裕 `E = T_total - S` を求める。
     - 余裕がある場合は、縮小済みの項目（特に `dontShrinkPriority == p` のグループ）について再計算を行い、必要であれば縮小量を減らして時間を戻す（あるいは通常の拡張ロジックで `E` を配分する）。

4. 上記の縮小→省略→再計算の流れを `p = 5` から `1` まで順に実行します。すべて試しても `D > 0` の場合は調整不能としてユーザに通知します（手動調整が必要）。

- 実装メモ: `shrinkStrategy` を設定可能にすると運用が柔軟になります（例: `"proportional"` / `"sequential"`）。

-- アンダー（`S < T_total`）のとき（余裕 `E = T_total - S`）: 拡張は `extendPriority` の高い順または比率的に行います。比率的拡張の JavaScript 実装例:

```javascript
// Wext: 拡張重み配列
const Wext = programs.map(p => p.extendPriority * Math.max(0, p.max - p.current_time));
const sumWext = Wext.reduce((a, b) => a + b, 0);
if (sumWext > 0) {
  for (let i = 0; i < programs.length; i++) {
    const pr = programs[i];
    const extPotential = Math.max(0, pr.max - pr.current_time);
    const W = Wext[i];
    const delta_ext = Math.min(extPotential, E * (W / sumWext));
    pr.current_time += delta_ext;
    E -= delta_ext;
    if (E <= 0) break;
  }
}
```

注: 上記は運用上の選択肢（`shrinkStrategy`、`skipThreshold` の設定、`skipPriority` の閾値など）で細かく制御できます。

この方法は「優先度レベルごとに縮小→省略→再評価」を行うことで、ユーザの希望通りにまず `dontShrinkPriority=5` を可能な限り縮め（または保護）、その次に `skipPriority=5` を検討するといった挙動を実現します。

### 運用モードと時刻表運行支援

#### 絶対時刻モード / 相対時刻モード
- `timeMode` を `"relative"`（相対）または `"absolute"`（絶対）で指定できるようにします。
  - 相対モード: 現在の仕様どおり「総時間 (`T_total`) の中で順に割当」して調整を行います。
  - 絶対モード: 各プログラムに `scheduledStart` / `scheduledEnd`（例: `"09:00"`）を持たせ、実時計（端末クロック）に合わせて運行します。絶対モードでは「時刻表を優先する（hard anchor）」か「柔軟に補正する（soft anchor）」かの運用フラグを用意すると良いです。

#### 絶対モード時の挙動（例）
- 指定時刻より前の場合は待機・カウントダウン表示。
- 遅延が発生した場合は、後続プログラムの `dontShrinkPriority` / `skipPriority` を使って縮小/省略してキャッチアップを試みるが、運用ポリシー（hard/soft anchor）により優先度を変える。
- 指定時刻を守るために、縮小の適用対象を「現在時刻からそのプログラムの開始までにある可変領域」に限定すると、重要プログラムの開始時刻を保障しやすい。

#### 時刻表運行支援機能案
- プログラムごとの `scheduledStart`/`scheduledEnd` のパース（日時/時刻の内部表現に変換）
- 遅延アラート（例: 遅延が閾値を超えたら黄色→赤の通知）
- 余裕があるときにのみ実施する「任意プログラム」を扱う仕組み（以下の `runIfTimeLeft`）
- 運行ログ（開始/終了実績）を保存して、次回のスケジュール改善に利用

#### デフォルトで省略するが時間が余れば実施するプログラム
- ユーザー提案どおり「省略優先度6」に相当する運用をサポートします。実装的には2つの扱い方を併用できます:
  1. `skipPriority: 6` を予約値として扱う（`skipPriority` の最大値を 6 とする）
  2. もしくは明示的なブールフラグ `runIfTimeLeft: true` を用意する（推奨）

 どちらの場合も基本方針:
  - 調整フェーズ（縮小・省略）では `skipPriority=6`/`runIfTimeLeft=true` の項目は最終段階まで無視して省略候補に含めない。
  - すべての縮小・省略・拡張が終わった後に余裕 `E = T_total - sum(current_time)` が残っていれば、`runIfTimeLeft` 項目に対して順番どおり（または優先度で）時間を配分して実施する。

 例 — `runIfTimeLeft` を反映する簡単な JavaScript: 

```javascript
// E: 残余時間（秒）
const deferred = programs.filter(p => p.runIfTimeLeft || p.skipPriority === 6);
for (const p of deferred) {
  const want = Math.min(p.baseline || 0, p.max || Infinity);
  const give = Math.min(want, E);
  p.current_time = give;
  E -= give;
  if (E <= 0) break;
}
```

#### 実装上の注意
- `scheduledStart` / `scheduledEnd` はローカル時刻とタイムゾーンを正しく扱う（または運用範囲をローカルタイムに限定する）。
- 絶対モードでは「開始時刻維持（hard anchor）」と「柔軟に補正（soft anchor）」の2モードを明示し、UIで切替可能にするのが実用的。
